### Step 1: Create `useEventListener` in `packages/core`

**Create `packages/core/src/use-event-listener.ts`:**

```ts
import type { HostElement } from './host-element.ts'
import { onMount } from './on-mount.ts'

/*
 * Attaches a DOM event listener to a host element. The listener is added when
 * the element connects to the DOM and removed when it disconnects. Returns a
 * dispose function for manual cleanup.
 */
export function useEventListener<K extends keyof HTMLElementEventMap>(
  host: HostElement,
  type: K,
  listener: (event: HTMLElementEventMap[K]) => void,
  options?: boolean | AddEventListenerOptions,
): VoidFunction {
  return onMount(host, () => {
    host.addEventListener(type, listener, options)
    return () => {
      host.removeEventListener(type, listener, options)
    }
  })
}
```

### Step 7a: Create `usePress` in `packages/utils`

Extract the press logic into a reusable `usePress` hook in `packages/utils/src/use-press.ts`. This encapsulates the click + keydown pattern so any component that needs "press" semantics can use it.

**Create `packages/utils/src/use-press.ts`:**

```ts
import { useEventListener, type HostElement } from '@aria-ui-v2/core'

/*
 * Attaches "press" interaction to a host element, handling both pointer and
 * keyboard input. The callback fires on:
 *
 * - `click` events (covers mouse, touch, and assistive technology)
 * - `keydown` events for Enter and Space keys (keyboard navigation)
 *
 * The keydown handler is needed because non-button elements (e.g. custom
 * elements with role="button") do not natively fire `click` on Enter/Space.
 * This follows the WAI-ARIA Button Pattern:
 * https://www.w3.org/WAI/ARIA/apg/patterns/button/
 */
export function usePress(
  host: HostElement,
  handler: () => void,
): VoidFunction {
  const dispose1 = useEventListener(host, 'click', handler)

  const dispose2 = useEventListener(host, 'keydown', (event) => {
    /* Ignore key events during IME composition (e.g. Chinese, Japanese input
     * methods). The composing state means the user is building a character
     * and Enter/Space should be handled by the IME, not by us. */
    if (event.isComposing) return

    if (event.key === 'Enter' || event.key === ' ') {
      event.preventDefault()
      handler()
    }
  })

  return () => {
    dispose1()
    dispose2()
  }
}
```

**Update `packages/utils/src/index.ts`** to export `usePress`:

```ts

```

### Step 7b: Rewrite `packages/elements/src/popover/popover-trigger.ts`

The current code uses `useInteraction` with the `press` interaction. Replace it with the new `usePress` utility.

**Before:**

```ts
import { computed, defineProps, useEffect, useInteraction, type Store } from '@aria-ui-v2/core'
import { press } from '@remix-run/interaction/press'

// ...

useInteraction(host, {
  [press]() {
    const store = getStore()
    if (!store) return
    if (!getDisabled()) store.emitOpenChange(!store.getOpen())
  },
})
```

**After:**

```ts
import { computed, defineProps, useEffect, type Store } from '@aria-ui-v2/core'
import { usePress } from '@aria-ui-v2/utils'

// Remove: import { press } from '@remix-run/interaction/press'

// ...

usePress(host, () => {
  const store = getStore()
  if (!store) return
  if (!getDisabled()) store.emitOpenChange(!store.getOpen())
})
```

**Why `click` instead of `pointerdown`/`pointerup`:** The `press` interaction dispatched on the full pointerdown-then-pointerup cycle on the same element, which is exactly what the native `click` event does. Using `click` is simpler and also correctly handles accessibility — screen readers and assistive technologies synthesize `click` events. The `keydown` handler covers Enter/Space for keyboard navigation (matching WAI-ARIA button patterns).

**What we lose (intentionally):**
- Long press support — not used by `PopoverTrigger`.
- Reentry protection / AbortSignal — the handler is synchronous, so not needed.
- `pointerleave` cancellation — `click` already handles this (no click fires if pointer leaves before release).

## Behavioral differences to be aware of

| Behavior | `@remix-run/interaction` press | Native `click` + `keydown` |
|---|---|---|
| Pointer click | pointerdown → pointerup dispatches `press` | `click` fires on mouseup inside element |
| Enter/Space | keydown (non-repeat) → keyup dispatches `press` | `keydown` fires immediately (no wait for keyup) |
| Long press | 500ms hold dispatches `longPress` | Not supported (not needed) |
| Pointer leaves during press | `pointerleave` cancels, no press | `click` doesn't fire (same result) |
| Escape during press | Dispatches `pressCancel` | Not supported (not needed) |
| Async reentry protection | Signal aborted on re-invocation | Not supported (not needed — handlers are sync) |

The key behavioral difference is that `keydown` fires on key-down rather than key-up. For a popover toggle this is acceptable and arguably more responsive. If key-up semantics are ever needed, the `keydown` handler can be changed to `keyup`.
